# 系统架构

## 目标

为广东本科普通批考生提供：**位次换算 → 硬性筛选 → 冲稳保推荐 → 志愿排序 → 自然语言解释** 的完整链路。

## 分层设计

### 1. 小程序层 (`miniprogram/`)

- **首页**：说明 + 开始填报
- **信息采集页**：分数、选科、偏好（城市/专业/院校层次）
- **结果页**：冲/稳/保志愿表 + 每条理由摘要
- **对话页**（可选）：基于已有草案追问调整

小程序只做展示与采集，**不做录取概率计算**（全部走后端 API）。

### 2. API 层 (`server/`)

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/v1/rank/lookup` | POST | 分数 → 位次 |
| `/api/v1/recommend` | POST | 生成冲稳保志愿草案 |
| `/api/v1/explain` | POST | 对单条志愿生成解释（LLM） |
| `/api/v1/chat` | POST | 多轮追问（LLM + 工具调用） |

### 3. 规则引擎（核心，非 LLM）

```
输入: province=广东, score, subjects, preferences
  ↓
位次换算 (score_rank 表)
  ↓
硬性筛选 (选科要求、批次、计划人数>0)
  ↓
冲稳保分档 (近3年 min_rank 与考生位次比较)
  ↓
偏好打分排序 (城市、专业、院校层次)
  ↓
输出: 45 个院校专业组建议序（可配置）
```

### 4. Agent 层（辅助）

LLM 只负责：

- 将结构化结果翻译为家长/考生能懂的话
- 回答「为什么推荐/为什么不推荐」
- 根据用户追问调整偏好后 **重新调用** `/recommend`

LLM **禁止**凭空编造录取分数或位次；解释时必须引用 API 返回的 `evidence` 字段。

## 部署建议

| 组件 | 推荐方案 |
|------|----------|
| 小程序 | 微信原生 + 云托管/自建 HTTPS API |
| 后端 | FastAPI + Docker，部署到腾讯云轻量/CVM |
| 数据库 | MVP 用 SQLite；上线换 PostgreSQL |
| LLM | 国内合规 API（通义/文心/DeepSeek 等），服务端调用 |
| 缓存 | Redis（可选，缓存一分一段查询） |

## 数据更新流程

每年 6 月高考后：

1. 下载广东一分一段表 → `score_rank`
2. 下载招生计划 → `enrollment_plan`
3. 整理往年录取 → `admission_history`
4. 运行校验脚本，对比样本考生人工结果
5. 更新 `data_year` 配置并发布
